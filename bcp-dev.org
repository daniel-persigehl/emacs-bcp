* Package Overview :world:
The BCP — Biblical Commentary and Prayer framework is an Emacs environment
for liturgical prayer and Bible study. It fetches scripture from configurable
backends (default: Oremus Bible Browser) or local files, renders Daily Office
liturgies, and supports annotated Bible study with org-roam cross-references.

** Design Principles :constraint:
- Rubric texts are NEVER altered — only shown/hidden
- Prayer/canticle texts are stored constants, never generated
- Variant forms are separate named constants
- Buffer is text-mode, read-only, overlays for all styling
- Spacing normalised post-render by bcp-liturgy--normalise-spacing

* Domain Architecture :world:
:PROPERTIES:
:STATUS: SETTLED
:END:

The codebase is divided into four domains. The boundary rule is:
if a module requires liturgical machinery to make sense, it belongs
under =bcp-liturgy-=. If it is genuinely reusable outside a liturgical
context, it belongs under =bcp-=.

** =bcp-= Shared Framework
Infrastructure with no domain semantics. Serves all other domains.
- Pure computus (Easter, dominical letter, moveable feasts)
- Western rank taxonomy
- API-agnostic fetch layer (backends register via bcp-fetcher-register-backend)
- Buffer rendering primitives
- User preferences

** =bcp-liturgy-= Liturgical Office (generic)
The generic Office machinery shared across all rites.
Not tied to any specific prayer book edition.
- Generic ordo walker and renderer
- Canticle registry (texts only meaningful in an ordo context)
- Propers protocol (the standard plist shape all rites return)
- Rite implementations live under their own sub-prefix:
  - =bcp-1662-=              Book of Common Prayer 1662
  - =bcp-1928-=              American BCP 1928
  - =bcp-roman-=             Roman Office (DA 1911 is test case)
  - =bcp-roman-lobvm-=       Little Office of the BVM
  - =bcp-roman-1962-=        Breviarium Romanum 1962 (future)
  - =bcp-roman-1568-=        Breviarium Romanum 1568 (future)
  - =bcp-roman-sarum-=       Use of Sarum (future)
  - =bcp-roman-monastic-=    Monastic Breviary (future)
  - =bcp-anglican-breviary-= Anglican Breviary 1955 (future)
  - =bcp-divine-worship-=    Divine Worship: Daily Office (future)

** =bcp-reader-= Scripture Reader
Simple, fast text display for study reading.
Passage navigation, translation management, local file loading.
No ordo, no appointed texts — just "show me this passage."

** =bcp-notebook-= Annotation and Cross-Reference
org-roam nodes, marginal note capture, backlink browser,
contextual views, spanning psalm editor, export.
Can in principle attach notes to any displayed text —
Office lessons, collects, reader passages alike.

* bcp-fetcher.el :science:
:PROPERTIES:
:STATUS: SETTLED
:END:

API-agnostic fetch dispatch layer. No dependencies on bible-commentary or bcp-1662.
Must be required first. Auto-loads bcp-fetcher-oremus at the bottom.
Sources register themselves via bcp-fetcher-register-backend with
`:kind :bible' (a Bible-wide backend) or `:kind :psalter' (Psalms-only).
The active Bible backend is `bcp-fetcher-backend' (default: 'oremus);
the active psalter is `bcp-fetcher-psalter' (default: 'coverdale).
When a Psalms passage is requested and a psalter is set, the psalter
is tried before the Bible backend.

** Backend Protocol
A source is registered as:
  (bcp-fetcher-register-backend 'NAME
    :name "Display Name"
    :kind :bible                      ; or :psalter (default :bible)
    :fetch-fn #'my-backend--fetch     ; (fn passage translation callback)
    :translations '("KJVA" "NRSV" …)) ; for completing-read and matching

The :fetch-fn must call (callback text) where text is a propertized string
(verse numbers formatted for visual-line-mode) or nil on failure.  Psalters
(`:kind :psalter') typically return nil for non-Psalms passages.

** Public API
- bcp-fetcher-fetch(passage callback &optional translation)
  Async. Dispatches to active backend. PRESERVES text properties.
  Do NOT clean result — would strip wrap-prefix/display properties.
- bcp-fetcher-fetch-passage(passage load-fn &optional translation)
  Async. Calls load-fn(text label). Used by bcp-reader / bcp-notebook.
- bcp-fetcher-fetch-passage-context(ref translation load-fn banner-fn)
  Context-window fetch for omitted/disputed verses. Banner-fn called
  immediately with status string; load-fn called with text + label.
- bcp-fetcher-clean-text(text)
  Cleans Windows-1252 C1 chars and nbsp. Text properties preserved.
- bcp-fetcher-register-backend(name &rest plist)
  Register a fetch backend. See Backend Protocol above.
- bcp-fetcher-available-translations()
  Returns translation list from the active backend (for completing-read).
- bcp-fetcher-active-translation(ref)
  Returns correct translation for REF per priority rules.
- bcp-fetcher-textual-status(ref), bcp-fetcher-textual-header-annotation
  Textual-critical status for omitted/disputed verses.

** Critical Constraint :constraint:
bcp-fetcher-clean-text and replace-match strip ALL text properties.
For Office buffer, DOM results returned DIRECTLY without cleaning.
In-buffer char cleaning in insert-text-block handles survivors.

** Cache Constraint :constraint:
Passage cache keys are (passage . translation). Changing psalm translation
or backend via the transient menu must clear the cache (bcp-fetcher-clear-cache)
to prevent stale results from the fallback chain being served.

** Translation-Aware Routing :constraint:
bcp-fetcher-fetch checks if the requested translation matches any registered
backend's :translations list (via bcp-fetcher--backend-for-translation).
If a match is found and it differs from the primary backend, that backend is
tried first in the fallback chain. This means setting psalm-translation to
"Vulgate" automatically routes psalm fetches to the vulgate backend without
requiring a change to bcp-fetcher-backend.

** Fallback Chain :constraint:
bcp-fetcher-fetch builds a list of (FETCH-FN . TRANSLATION) attempts, tried
in order until one returns non-nil:
0. Active psalter (only when passage is a Psalm AND either no explicit
   translation was passed, or the caller's translation is among the
   psalter's :translations)
1. Translation-matched backend (if different from primary and psalter)
2. Primary backend with preferred translation
3. Primary backend with `bcp-fetcher-fallback-translation' (default: KJVA)
4. Fallback backend (`bcp-fetcher-fallback-backend') with preferred translation
5. Fallback backend with fallback translation

Step 5 is the critical rescue for the case where the preferred translation
is served by a primary backend that can't handle the passage (e.g. the
vulgate backend is psalter-only, so non-psalm lessons return nil even though
the translation matches), AND the fallback backend doesn't recognise the
preferred translation (e.g. Oremus has no Vulgate version code).  Without
step 5, LAT profile + non-psalm lesson = unrecoverable fetch failure.

** Render Fallback Button :ui:
When a passage fetch ultimately returns nil, Office renderers call
=bcp-liturgy-render--insert-passage-fallback= (in bcp-liturgy-render.el)
instead of emitting raw text.  This inserts a shadow-faced text-button
reading "[LABEL — fetch failed; click to retry]".  Activating the button
re-invokes =bcp-fetcher-fetch-passage= and, on success, replaces the button
region with the fetched passage text; on repeated failure the button is
reinstalled.  Buttons are live even when the Office buffer is read-only
(the retry handler temporarily binds `inhibit-read-only').

Prior behaviour emitted literal org-link syntax ("[[bible:...][...]]") into
text-mode buffers, which rendered as unusable raw characters — the original
fallback assumed an org-mode Office buffer that was later rejected.

** Furigana Display System
Four display modes controlled by =bcp-fetcher-furigana-display=:
- =normal= — furigana displayed inline with body text
- =comment= — furigana displayed in muted grey (font-lock-comment-face)
- =hidden= — furigana invisible (Emacs =invisible= property with =bcp-furigana= spec)
- =rubi= — overhead rubi via inline SVG images (kanji + reading kana centred above)

=bcp-fetcher--propertize-furigana= marks 《…》 spans with the =bcp-furigana=
text property at render time, and stamps =bcp-rubi-reading= / =bcp-rubi-base-width=
on the first character of each preceding kanji run so the rubi renderer can
locate spans at finalise time.

*** Rubi SVG renderer
=bcp-fetcher--rubi-svg-build= generates an SVG per (kanji . reading) pair
sized to the current =frame-char-width= / =frame-char-height=.  Tunables:
- =bcp-fetcher-rubi-svg-kanji-scale= (default 0.62) — kanji height as a
  fraction of =frame-char-height=
- =bcp-fetcher-rubi-svg-reading-scale= (default 0.5) — reading height as a
  fraction of the kanji size; auto-shrinks further when the reading would
  otherwise overflow the kanji-width box
- =bcp-fetcher-rubi-svg-font= — font-family list passed to the SVG
Built images are cached in =bcp-fetcher--rubi-svg-cache= (keyed by cons
cell), invalidated by =bcp-fetcher-clear-cache=.
=bcp-fetcher--rubi-apply-svg= walks the buffer, replacing each kanji
span's display with its SVG image; =bcp-fetcher--rubi-remove-svg= strips
those =display= / =bcp-rubi-svg= properties.  When SVG is unavailable,
=rubi= mode falls through to showing the 《…》 markers visibly inline.
Diagnostic: =M-x bcp-fetcher-rubi-svg-diagnose= prints the current pixel
sizing.

*** Toggle semantics
=bcp-fetcher-toggle-furigana= is a binary "show-as-configured ↔ hidden
entirely" toggle.  State is tracked via buffer-local
=bcp-fetcher--furigana-toggled-off=; =bcp-fetcher--furigana-show-configured=
dispatches on =bcp-fetcher-furigana-display= to restore the configured
mode (including re-applying SVGs for =rubi=), and
=bcp-fetcher--furigana-hide-all= strips SVGs and stamps every 《…》 span
invisible.  The Advanced override menu selects *which* mode shows when
toggled on; the main-menu toggle only flips visibility.

The command searches for a buffer with furigana annotations (current
buffer preferred, otherwise the first live buffer with =bcp-rubi-reading=
or =bcp-furigana= properties), so it works when invoked from the settings
transient regardless of which buffer was selected beforehand.

The language profile auto-promotes furigana from =hidden= to =rubi= when
the backend is =bungo-yaku= but the profile is not JAP (cross-language override).

* bcp-profile.el :science:

Language profile system.  A profile sets all scripture and liturgical language

** Profiles
| Profile | Lesson | Psalm | Backend | Fallback | Roman lang | Canticle lang | Gloria | Furigana |
|---------+--------+-------+---------+----------+------------+---------------+--------+----------|
| ENG | KJVA | Coverdale | oremus | oremus | english | english | nil | hidden |
| ENG-TB | KJVA | Tate & Brady | oremus | oremus | english | english | nil | hidden |
| LAT | Vulgate | Vulgate | oremus | oremus | latin | latin | t | hidden |
| JAP | Bungo-yaku | Bungo-yaku | bungo-yaku | oremus | english | english | nil | rubi |

The active psalter (=bcp-fetcher-psalter=) is not a profile field: it is
derived from the psalm translation at apply time via
=bcp-fetcher--psalter-for-translation=.  Picking a translation that a
registered =:kind :psalter= backend serves ("Coverdale", "Tate & Brady",
"Vulgate", etc.) routes psalms through that psalter; any other name
leaves psalter at nil and the Bible backend serves psalms directly.

** Architecture
=bcp-profile--definitions= (defconst) maps profile symbols to keyword→value plists.
=bcp-profile--setting-to-variable= maps keywords to the real variables they govern.
=bcp-profile--setting-to-override= maps keywords to override defcustom symbols.

=bcp-profile-apply= resolves each setting: if the override is ='default=, use the
profile's value; otherwise use the override.  Writes to real variables, so existing
code reads them unchanged (no interception layer).  After writing, it derives
=bcp-fetcher-psalter= from the effective psalm translation.

** Override Variables
All default to ='default= (inherit from profile):
- =bcp-profile-lesson-translation= → =bible-commentary-translation=
- =bcp-profile-psalm-translation= → =bible-commentary-psalm-translation= (also drives =bcp-fetcher-psalter=)
- =bcp-profile-backend= → =bcp-fetcher-backend=
- =bcp-profile-fallback-backend= → =bcp-fetcher-fallback-backend=

*** Japanese Anglican psalm chanting — research notes
Source: NSKK 聖歌集改訂委員会 newsletters №2 (1999), №5 (2000), №7 (2001),
at =www.nskk.org/hymn/pdf/002.pdf=, =005.pdf=, =007.pdf=.

The NSKK chants psalms and canticles at the Offices using three
stacked traditions, all provided in the 2006 聖歌集 service-music section
(numbers S41–S70, 礼拝諸式・詩篇):
1. *Gregorian psalm tones* (プレーンソング詩篇唱) — the eight church modes,
   with the classic intonation → reciting tone → mediant → reciting tone →
   termination shape.
2. *Anglican chant* (アングリカンチャント) — described by the committee as
   "plainsong with harmony added", pointed verses with a mediant dividing
   the verse into two halves, 4-part harmony.
3. *Original Japanese compositions* — new tones composed for Japanese
   prosody.

Newsletter №2 contains a specimen Mode II Easter canticle (復活の歌) laid
out in the classic Gregorian column format: Japanese syllables aligned to
intonation, mediant cadence, termination.  So *the mediant concept
transfers directly to Japanese*; it just points Japanese syllables to the
tone instead of English ones.

Committee observation: psalm tones fit Japanese *better* than fixed
metrical hymn tunes, because the elastic reciting tone absorbs extra
syllables that wouldn't scan in a metrical setting.  Anglican chant is
harder to adapt (fixed note count per half-verse) and they noted difficulty.

Canticle taxonomy (shared with Latin): 大カンティクル (Magnificat, Nunc
dimittis, Benedictus) vs. 小カンティクル (other OT canticles).

**** Conventions
- *Versification:* Hebrew/KJV, matching =bcp-liturgy-bungo-yaku.txt=
  (not Coverdale BCP numbering).  Note Bungo verse merges, e.g.
  Ps 1:3 Bungo = Coverdale 3+4.
- *Mediant:* =＊= (U+FF0A fullwidth asterisk).  Half-width =*= is
  visually cramped against CJK text; fullwidth gives the singer a
  prominent cue without breaking rubi alignment (rubi SVGs are
  anchored per-kanji via =bcp-rubi-base-width=, independent of
  surrounding glyph widths).
- *Flex:* =†=.  Used only on long composite verses.  Single mediant
  is the default; a flex is a genuine secondary breath, not padding.
- *Placement rule:* =＊= marks the *Japanese* breath point, not the
  position where English =*= falls.  Anchor conceptually to Hebrew
  parallelism (A // B) via the pre-existing Coverdale/Vulgate
  pointing, then find the Japanese clause boundary carrying the same
  parallelism hinge.  Natural Japanese breath markers: conjunctive
  =て=, negative =ず=, topic =は=, conditional =ば=, verb endings in
  =たまふ=.  Japanese word order often inverts the English, so the
  mediant position diverges — that is correct, not a bug.
- *Stanza headings:* (Ps 119) inline at start of verse 1 of each
  stanza, preserving the Bungo source's style.
- *Furigana:* preserved as =kanji《reading》= from source.
- *Spacing:* single ASCII space on either side of =＊= and =†=.

Architectural commitments (user-confirmed):

*** Metrical psalters are hymns
A metrical psalter entry reuses the hymnal file format, registry, and
renderer.  Psalm identity is carried by the psalm number; each metrical
setting is a hymn keyed to that psalm (and to a translation/tune).  This
avoids inventing a new file type and avoids verse-numbering conflicts
with the prose psalter — a metrical setting renders as a hymn block
(stanzas, meter, tune), not as numbered verses.

*** Office substitution via the hymn renderer
Metrical psalms may stand in for prose psalms at the Office psalm slot.
Dispatch: when rendering a psalm, if the user's preferred psalter for
the current language is metrical, call the hymn renderer with the
corresponding setting.  No verse-numbering shim needed because the
output is a hymn, not a numbered psalter line.

*** Preferred-psalter setting is one dimension, not two
No separate "prefer metrical" toggle.  The language profile's preferred
psalter directly names the psalter (Coverdale, Tate-Brady, Anglo-Genevan
1987, Japanese Genevan, etc.).  If that psalter is prose-pointed, the
pointed renderer runs; if metrical, the hymn renderer runs.  One
setting, unambiguous dispatch.

*** Tunes are first-class and shared
Tunes carry meter and optional source metadata, independent of any
specific psalter text.  The Genevan tune corpus is referenced by both
the (English) Anglo-Genevan 1987 and the Japanese Genevan settings; one
tune record serves both.  Sets up future audio hooks cleanly.

*** Rubi/furigana applies
Japanese metrical psalters (Japanese Genevan) route through the same
rubi SVG infrastructure as other CJK translations.  No new rendering
code required; the hymn renderer must preserve =《...》= markup.

*** Private data directory
=defcustom bcp-private-data-dir= (nil by default), set in the user's
Emacs config.  Fetcher lookup order: private dir → shipped data dir.
Registry entries for copyright-encumbered psalters (Anglo-Genevan 1987,
Japanese Genevan, etc.) declare expected filenames; if a file is absent
in both locations and the entry is marked private-only, surface a clear
"install your licensed copy at X" message.  Never commit copyrighted
text to the repo.

**Backend wiring (2026-04-23):** Integrated via the dedicated psalter
layer (=:kind :psalter= on =bcp-fetcher-register-backend=, selected by
=bcp-fetcher-psalter=).  Renderer lazy-loads the text file into a
hash-table keyed by psalm number, each psalm stored as a list of
"blocks" (part heads and stanza groups), and serves a verse range by
keeping every stanza whose verse numbers overlap the range.  Stanzas
are the atomic unit — you can't sing half a quatrain — so a narrow-
range query may return a slightly wider span.  Verse numbers on
rendered text carry =bcp-verse/bcp-book/bcp-chapter= text properties
for downstream highlighting.  Translation names: "Tate & Brady" / "T&B".
A new language profile =ENG-TB= selects =oremus= as backend plus
Tate & Brady for psalms, with KJVA lessons.  In the transient menu
the psalter is derived from the "Psalms" translation choice, not
selected separately.
Post-ingest text-level fixups handled (a) part-label renumbering (TCP
transcription anomalies in Pss 37, 89, 119), and (b) verse-1 injection
on ~27 psalms where the TCP dropped the opening verse milestone.  Pss
69 and 71 open with a metrical stanza covering prose verses 1–2;
labelled as "1" so full-psalm queries work, narrow "verse 2 alone"

a psalter attempt ahead of the backend chain when (a) the passage is
a Psalm, (b) a psalter is active, and (c) the caller either passed no

**Psalter derived from translation (2026-04-23):** The profile layer

** Architecture (2026-04-23)

Three normalized registries live in =bcp-hymnal.el=:

- =bcp-hymnal--texts= — text-id → plist.  One record per
  translation/original pair; a Latin hymn with three English renderings
  is four records.  Lazy-loaded from =bcp-hymnal-texts.txt= (shared
  across hymnals).  Other authoring surfaces (e.g.
  =bcp-roman-hymnal.el=) emit synthetic records into this hash via
  =bcp-hymnal-register-exporter= — see /Roman hymnal export/ below.
- =bcp-hymnal--tunes= — tune-id → plist.  One record per canonical
  tune (NICAEA, STUTTGART, SINE-NOMINE, …).  Meter and composer
  attached here, not on the slot.
- =bcp-hymnal--manifests= — hymnal-id → hash table keyed by hymn
  number (string, may include letter suffix: "164b").  The per-hymnal
  number → slot-plist mapping — the only thing that differs between
  books that share a text/tune.

Copyright attaches to the component that is or isn't copyrighted
(text / translation / tune / harmonization / arrangement), not to
the slot.  Slot entries can override with a local note.

** Selection

Three orthogonal axes the user controls:

1. =bcp-hymnal-available= — list of enabled hymnal ids.
2. =bcp-hymnal-office-profile= — slot profile for the office
   (=anthem-only= / =opening-and-close= / =opening-full= /
   =roman-office= / =monastic-office= / =none=).  Defaults to
   =auto=, which reads the language profile: LAT → =roman-office=,
   ENG / ENG-TB (strict 1662) → =anthem-only=, ENG-1928 →
   =opening-full=.
3. Selection policy per slot (=appointed= / =sticky= / =suggest= /
   =fixed=) — not yet wired into the renderers.

Translator preference (=bcp-hymnal-preferred-translator= with a
fallback list: Neale → Caswall → Newman → Chambers → Riley → Primer
→ Britt) picks between multiple English renderings of the same Latin
office hymn.

** Hymnal 1940 (2026-04-23)

Scraped 866 entries from Hymnary.org (=HPEC1940=; 1 req/sec, UA
identified as research client).  Metadata → Elisp by
=/tmp/hymnary_to_elisp.py=:

- =bcp-hymnal-tunes.el= — 642 distinct tunes (public-domain filter
  based on composer / harmonizer /arranger years ≥ 1931 as of 2026).
- =bcp-hymnal-1940.el= — manifest; each slot carries
  =:first-line :tune :meter :author :translator :topic :copyright=.
- =bcp-hymnal-1940-manifest.txt= — human-readable audit; lists the
  103 restricted slots with which component is blocked.

763 of 866 slots are currently available; 103 flagged as restricted
because a contribution (text / translation / tune / harmonization)
first appeared 1931 or later and has not yet entered PD under the
95-year rule.  The compilation itself is still under copyright
through ~2035, but Church Publishing's Rights & Permissions page
states that selections without a copyright listing are PD; we take
them at their word for the 763 unencumbered slots.

Stanza bodies are not in this metadata scrape — only first lines.
A second pass (see /Stanza ingestion/ below) fetches them from the
canonical /text/<slug>/ pages and merges them into
=bcp-hymnal-texts.txt=.

** Tag system (2026-04-23)

Seeded the selection infrastructure in =bcp-hymnal.el=.  Four pieces:

*** Controlled tag vocabulary
=bcp-hymnal--tag-axes= groups tag symbols by axis — season, occasion,
hour, slot, character, mood, theme.  A symbol not listed in any axis
is rejected (grow-on-demand discipline).  Tags attach to text records,
not slots: the character of a hymn is a property of the text, not of
which hymnal prints it.

*** One-directional implication graph
=bcp-hymnal--tag-implications= encodes specific → general
relationships (=low-sunday= → =(easter-octave eastertide resurrection
paschal)=, =whitsunday= → =(pentecost-octave holy-spirit paraclete)=,
etc.), expanded at match time by =bcp-hymnal--expand-tags=.  Authors
tag conservatively on the most-specific label; expansion broadens the
match surface without bloating storage.

*** Date → occasion translator
=bcp-hymnal-occasions-for-date (DATE)= returns a specificity-ordered
list, most specific first:

#+begin_example
Easter Day 2026-04-05:     (easter-day easter-octave eastertide)
Easter Monday:             (easter-monday easter-octave eastertide)
Low Sunday 2026-04-12:     (low-sunday easter-octave easter-week-1 eastertide)
Mon after Low Sunday:      (easter-week-1 eastertide)
Good Friday 2026-04-03:    (good-friday holy-week passiontide)
Christmas Day:             (christmas-day christmas-octave christmastide)
All Saints:                (all-saints trinitytide)
#+end_example

Week numbering aligns with 1662 (Low Sunday begins Easter Week 1);
the Easter Octave is Easter Sunday + 7 and does not carry a
=easter-week-N= tag.  Uses =bcp-moveable-feasts= + fixed-date feast
table + =bcp-1662-liturgical-season= under the hood.

*** Appointment policy + selector
=bcp-hymnal-appointment-policy= is a per-slot-kind alist with three
values: =appointed= (rubric wins, no picker — Pian rigorist),
=suggest= (appointed hits rank first, tier-matches follow — default),
=free= (ignore appointments entirely — liturgical libertine).  A
=default= key handles unlisted slot-kinds.

=bcp-hymnal-select= takes =:date :occasion :slot-kind :language
:extra-tags :policy= and returns text-ids ranked by:
appointment-status → occasion-tier → =popular= → alphabetical.
Occasion is a *ranking signal*, not a filter; the hard filters are
language, extra-tags, and (under =appointed= policy) appointment
status.  The sort puts rubric-appointed hits at the top, then
tier-0 matches, deeper tiers, and finally hymns with no occasion
match at all (tier 9999).

** Stanza ingestion (2026-04-25)

The "Stanza bodies are not in the scrape" gap noted under Hymnal 1940
is closed.  Pipeline:

1. =/tmp/hymnary_text_scrape.py= — visits each =/hymn/HPEC1940/<hid>=
   page, follows the canonical =/text/<slug>= link, extracts stanzas,
   copyright, and source attribution.  Resumable (state in the output
   JSON), polite (1.5s between requests), identifies as a research
   client.  Skips slots whose =:copyright (:restricted ...)= plist
   blocks =:text= or =:translation= — those have no usable PD text
   on Hymnary either.
2. =/tmp/hymnary_merge.py= — turns the scrape into ingested data:
   - Appends one =[text-id]= block per unique text to
     =bcp-hymnal-texts.txt=.  The text-id is the kebab of the first
     line; collisions (same opening words across distinct texts) fall
     back to the Hymnary slug.  Two slots that share a Hymnary slug
     are the same text bound to different tunes — they share a
     text-id (hymnal-level dedup).
   - Inserts =:text <id>= into each manifest slot in =bcp-hymnal-1940.el=,
     immediately after =:first-line=.  Idempotent: re-runs skip slots
     already linked.
   - Comments out slots whose =:copyright (:restricted ...)= plist
     gates the text or translation (no other component can save them).

End state: 826 hids scraped, 815 with stanzas, 585 unique text blocks
appended.  815 of the 866 manifest slots now resolve to a text record;
40 commented out as restricted; 11 already had =:text= from an earlier
POC seed.  Scripts left under =/tmp= (gitignored per project policy).

** Calendar-aware occasion translation (2026-04-25)

The hymn that's right for a date depends on /which calendar/ the
office is using.  =bcp-hymnal-occasions-for-date= now dispatches
through =bcp-hymnal--calendar-translators=.  Three built-ins:

- =anglican-1662= — the maximal occasion superset minus feasts not
  in the 1662 kalendar (Transfiguration, Assumption, Holy Cross,
  St Joseph, Sacred Heart, Corpus Christi, Christ the King, Visitation,
  St Mary Magdalene).
- =anglican-1928= — superset minus the smaller non-1928 set
  (Assumption, Sacred Heart, Corpus Christi, Christ the King, St Joseph).
- =roman-pre-1955= — superset plus Roman moveables not derivable from
  the 1662 framework: Corpus Christi (Easter+60), Sacred Heart
  (Easter+68 = Friday after Corpus Christi octave), Christ the King
  (last Sunday of October per /Quas primas/ 1925, before the 1969
  reform moved it).

=bcp-hymnal-calendar= defcustom: when set to =auto= (default), it
reads =bcp-language-profile= — LAT → =roman-pre-1955=, ENG-1928 →
=anglican-1928=, otherwise =anglican-1662=.  Users can register
custom kalendars via =bcp-hymnal-register-calendar= and bind
=bcp-hymnal-calendar= to their id.  =bcp-hymnal-select= takes
=:calendar= as a per-call override.

This satisfies the design rule: an Anglo-Catholic 1662 user with a
plain 1662 kalendar won't get a hymn for Pope Pius V, but the same
user with a custom kalendar that lists Pius V will.

** Renderer wiring (2026-04-25)

Hymn slots wired into the Anglican ordo renderer.  =bcp-anglican-render=
gained a =:hymn= step: it calls =bcp-hymnal-select= with the slot's
=:slot-kind= and =:extra-tags=, takes the top text, and renders its
first-line + stanzas under a "Hymn" heading.  When no candidate is
found the slot degrades to a plain rubric.

1662 Morning and Evening Prayer ordos and the 1928 ordo each gained
two =(:hymn :slot-kind ...)= insertions: =office-hymn= after the
opening responses, =closing= after the Grace.  Following the
=:office-profile= scheme, =anthem-only= profiles render the closing
slot as a plain rubric — the slot is present in every ordo but only
fills with a hymn when the user's profile asks for one.

=:text= record.  Runs as a load-time exporter from
